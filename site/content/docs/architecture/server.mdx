The server is a single Fastify app (`server/src/index.ts`). It owns the agent runtimes, exposes a REST + SSE API under `/api/*`, relays provider traffic for the Claude CLI, and serves the SPA. Everything runs from one process, bound to loopback by default.

## Configuration

All tunables live in `server/src/config.ts` and read from the environment, so the launchers and `start.sh` can steer behaviour without touching code.

## Boot Sequence

`server/src/index.ts` wires the app in a fixed order ā€” plugins, then routes, then cache warm-up, then listen.

<Steps>
<Step>
### Build the Fastify instance

A logger with a custom request serializer that **redacts `?token=` out of every logged URL** (share links would otherwise leak into logs), `ignoreTrailingSlash`, a 2 MB body limit (GenUI prompts grow), and `maxParamLength: 4000` so deep worktree paths encoded as route params don't 414.
</Step>
<Step>
### Arm auth

`resolveToken(HOST, AUTH_TOKEN)` returns a token; if the server is bound to a **non-loopback** host with no token set, it generates one and prints a `?token=ā€¦` connect string straight to stdout (never to the structured log). An `onRequest` hook enforces it.
</Step>
<Step>
### Register routes

Every `register*Routes` call mounts a slice of the `/api/*` surface ā€” sessions, settings, relay, codex, git, search, agents, terminal, and more. All under one encapsulated plugin scope.
</Step>
<Step>
### Serve the SPA

If `web/dist` exists, `@fastify/static` serves assets with `index: false` so the server can route `/` itself and pick the engine's SPA entry. In dev (`vite` on `:5273`) the dist is absent and the server runs API-only.
</Step>
<Step>
### Warm caches and listen

Settings, worktrees, permission rules, codex config, labels, schedules, and share caches warm before `app.listen`. After binding, it pre-warms the GenUI type-check and builds the search index in the background so the first real turn never pays those costs.
</Step>
</Steps>

## The Claude Turn Loop

`server/src/lib/claude-runner.ts` wraps the Claude Agent SDK. `runClaude` is an **async generator** that yields a `RunnerEvent` for every SDK message, which the route layer maps onto the SSE contract.

```ts
export type RunnerEvent =
  | { type: 'delta'; text: string }
  | { type: 'tool_use'; id: string; name: string; input: unknown }
  | { type: 'tool_result'; tool_use_id: string; text: string; isError: boolean }
  | { type: 'permission_request'; id: string; toolName: string; input: unknown }
  // ā€¦and more
```

Tool permissions flow through a `canUseTool` callback: when the SDK wants to run a tool that needs approval, the runner emits a `permission_request`, parks the turn, and waits for the client to POST a decision back before resuming. The SDK's default 60 s MCP-tool timeout is raised to **5 minutes** at process start (`CLAUDE_CODE_STREAM_CLOSE_TIMEOUT`) because a complex GenUI render can take 30ā€“120 s.

## The Provider Relay

The Claude CLI expects an Anthropic-shaped API. To point it at a third-party provider (Macaron, OpenRouter, LiteLLMā€¦) the server runs a reverse proxy at `/relay/anthropic/:providerId/v1/...` (`server/src/routes/relay.ts`).

<Callout type="info" title="Why a relay and not just a base URL swap">
  The CLI probes several `/v1/` endpoints at startup (models, org) to validate the session. If those 404, it aborts with a misleading "issue with the selected model" error ā€” even when `/v1/messages` would work. The relay **synthesizes** `/v1/models` and `/v1/models/<name>` from the active provider config, forwards `/v1/messages` verbatim (streaming byte-by-byte), and returns an empty `{ok:true}` 200 for every other probe.
</Callout>

Active provider config is persisted to `~/.claude/macaron-config.json` via `settings-store.ts` and edited from the Settings page.

## The SSE Contract

Every streaming surface ā€” starting a session, sending a message, tailing a live turn ā€” speaks one union type defined **once** in `shared/src/sse.ts` and consumed by both the server (publisher) and the client (reader).

<Accordions>
<Accordion title="Assistant + reasoning stream">
`delta` carries assistant prose; `reasoning` is the thinking stream, kept distinct so the client can collapse it as its own block. `user-text`, `starting`, and `meta` bookend a turn.
</Accordion>
<Accordion title="Tool lifecycle">
`tool_use` ā†’ `tool_input_delta` / `tool_input_done` (streamed args) ā†’ `tool_result`. Permission gates ride alongside as `permission_request` / `permission_resolved`.
</Accordion>
<Accordion title="Codex-native surfaces">
`codex_plan` (a monotonically-updated plan card) and `codex_approval_request` / `codex_approval_resolved` (command/file/network approvals) model Codex's own turn primitives ā€” see the [Codex Variant](/docs/architecture/codex) page.
</Accordion>
<Accordion title="Lifecycle + telemetry">
`usage` (output/thinking tokens), `event`, `log`, `warn`, `error`, and a terminal `done` with an exit code. `live-end` closes a tailing connection.
</Accordion>
</Accordions>

A separate `SystemEvent` stream (`GET /api/events`) pushes a debounced `sessions-changed` nudge whenever a transcript file changes on disk ā€” so sessions started in a plain terminal surface live in the UI instead of on the next slow poll.

## Reading Sessions From Disk

The WebUI does not own a database. `session-store.ts` reads the JSONL transcripts Claude Code already writes under `~/.claude/projects/**/*.jsonl` (and Codex rollouts under `~/.codex/sessions/**`), parses them into the shared session/turn types, and a `session-watcher` tails the trees for changes. This is why the WebUI shows sessions you started outside the browser.
